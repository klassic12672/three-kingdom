# Prompt Package: SP-04X2 Minimal Final Closeout

Use after SP-04X1 is accepted. This package proves the explicit SP-04 contract, measures integrated performance once, fixes only demonstrated gaps, and closes the plan.

## Authorization boundary

The user has authorized local SP-04 remediation, commits, and ordinary non-force pushes to the approved `origin/main`. Verify remote, branch, tree, and divergence before each commit or push.

No force push, rebase, branch, tag, pull request, release, workflow dispatch/rerun, signing, Steam, deployment, publishing, credential, or remote-setting action is authorized. Physical Windows remains M4; signing and Steam remain SP-15.

## Prompt

~~~text
/goal

Complete SP-04X2 — explicit-requirement audit, one integrated performance proof,
minimal remediation, exact-SHA hosted evidence, and final SP-04 closeout.

Ponytail routing

- Invoke `ponytail:ponytail` in full mode before defining remediation.
- Invoke `ponytail:ponytail-review` for the final over-engineering review if available.
- Classify every recorded limitation as required by SP-04, deferred, or owned by
  another plan. Fix only the first category.
- Do not turn closeout into implementation of SP-05, SP-08, SP-11, SP-12, SP-13,
  or earlier packages' explicitly declared non-goals.

Dependency gate

- Consolidated F9, G, and X1 must be accepted with exact-SHA evidence.
- X1 must have local revision-bound visual evidence.

Read AGENTS.md, MASTER_PLAN, ROADMAP, the complete SP-04 plan, applicable ADRs,
CHARACTERS, SIMULATION, CONTENT_PIPELINE, DEVELOPMENT, all accepted SP-04 evidence,
and the current implementation/tests/artifacts.

Use:

- project-architect, read-only, for requirement ownership and ADR review;
- verification-reviewer, read-only, for independent completion evidence;
- a specialist writer only for a concrete demonstrated blocker.

Phase 1: audit before editing

Build one concise matrix for:

- the SP-04 goal;
- its explicit requirements and public contracts;
- seven workstreams;
- listed edge cases, performance budgets, and tests;
- seven acceptance criteria;
- applicable locked MASTER_PLAN character rules.

For each row record evidence, status, and the smallest required fix. Do not expand
“every prior limitation” into scope. Deferred or externally owned items stay out.

Required proof

Prove or minimally remediate:

1. Historical/custom structured characters and persistence.
2. Distinct family, household, retinue, employer, and future political boundaries.
3. Nine relationship dimensions and bounded consequential memories.
4. Political marriage versus consenting adult romance.
5. Childhood betrothal and coercion safeguards.
6. Children, adoption, guardianship, education, death, succession, inheritance,
   and minimal regency behavior.
7. Deterministic simultaneous-death, missing-heir, extinct-family, captivity,
   stale-command, and disputed-succession behavior.
8. Player continuity with a valid successor and explicit no-successor behavior.
9. Knowledge-filtered public queries.
10. Minimal required battle capture/death/rescue/shared-memory round-trip.
11. Bilingual bounded content and the X1 inspector.
12. Save/load, recovery, replay, tier transitions, current forward migrations,
    checksums, stable IDs, canonical ordering, and bounded histories.

Do not require placeholder faction/court/office/title/diplomacy implementations;
the SP-04 rule is that these roles remain distinct and existing character events
provide later integration seams.

Performance

Use one integrated 1,000-character fixture that exercises realistic bounded
SP-04 state. Enforce only the documented budgets:

- routine integrated SP-04 processing under 3 seconds on the development Mac;
- one detailed character/social query under 100 ms;
- bounded memory/history growth.

Reuse existing fixture builders and performance tests. Do not create a second
benchmark framework or per-subsystem duplicates. Record machine, configuration,
fixture counts, repetitions, result, and checksum.

Verification

Run focused checks for any remediation, then:

./scripts/validate.sh
./scripts/test.sh Release
./scripts/import.sh
git diff --check
git lfs fsck

Also run the integrated performance test, current save/migration/recovery coverage,
local macOS export/smoke, and only X1 visual regressions affected by remediation.
The full suite already runs the soak; do not run a duplicate standalone soak
unless coverage analysis proves it is excluded.

Closeout

1. Obtain architecture, verification, and Ponytail reviews.
2. Resolve only validated SP-04 blockers.
3. If any explicit criterion remains unproven, leave SP-04 Active and report it.
4. Otherwise commit the final implementation/remediation, re-fetch, and
   ordinarily push `main`.
5. Observe automatic CI only and require exact-SHA hosted macOS/Windows tests,
   import, native export, smoke, manifests, and artifacts.
6. Download and inspect both final artifacts once.
7. Add the final completion matrix and exact-SHA evidence.
8. Only then mark SP-04 Complete and unblock SP-05, commit the closeout, and push.

Definition of done

- Every explicit SP-04 acceptance criterion has direct evidence.
- Integrated performance budgets pass without weakened thresholds.
- Exact-SHA hosted and final artifact evidence pass.
- Status/evidence are committed and pushed; the tree is clean.
- Deferred and externally owned systems were not pulled into SP-04.
~~~
